## [LRN-20260501-002] correction

**Logged**: 2026-05-01T00:00:00+08:00
**Priority**: high
**Status**: pending
**Area**: floating-layout

### Summary
Floating tray fixes must test startup timer races as well as direct merge calls.

### Details
A scripted delayed-collision tray test initially failed because `_restore_floating_tray_state()` was still scheduled after startup and cleared members generated by an early interaction. Directly calling `_on_card_interaction_finished()` was not enough to catch this; the realistic timer path was required.

### Suggested Action
For floating tray work, run a test that creates overlapping floating cards before and after the startup restore timer fires, then assert tray members remain visible and recoverable.

## [LRN-20260501-004] correction

**Logged**: 2026-05-01T12:08:00+08:00
**Priority**: critical
**Status**: pending
**Area**: visible-ui-validation

### Summary
HainTag tray/Dock fixes must include a visible app screenshot because offscreen assertions can miss duplicate surfaces.

### Details
The user showed a screenshot where Dock and the floating storage tray displayed the same cards at the same time. Offscreen tests checked state transitions but did not catch the visible duplication. The correct validation path is to restart the real GUI, capture the screen, and inspect that duplicated recovery surfaces are gone.

### Suggested Action
For visible state bugs, always add a real GUI launch plus screenshot capture after logic tests, especially when Dock, floating tray, right rail, or window persistence are involved.

## [LRN-20260501-003] correction

**Logged**: 2026-05-01T11:52:00+08:00
**Priority**: high
**Status**: pending
**Area**: floating-tray

### Summary
The HainTag storage tray should not remain visible when it no longer has enough content to justify being a separate surface.

### Details
The user corrected that a storage tray with no cards, or with only one remaining card, feels intrusive and should disappear. A single remaining stored card must still have a recovery route through Dock, so hiding the tray must not drop the member state.

### Suggested Action
For storage tray changes, verify the visible rule explicitly: zero members hidden, one member hidden with Dock recovery intact, two or more members visible, and restoring down from two to one hides the tray.

## [LRN-20260501-002] correction

**Logged**: 2026-05-01T08:02:30+08:00
**Priority**: high
**Status**: pending
**Area**: floating-tray

### Summary
The HainTag floating tray must be treated as a direct-manipulation UI surface, not just a passive restore list.

### Details
The user corrected that the tray was hard to use because it could not be dragged and floating cards could be unexpectedly swallowed by proximity. A usable tray needs real mouse drag handling, persisted position changes, full-row restore targets, and no surprise auto-store when cards merely overlap.

### Suggested Action
For tray changes, run Qt mouse-event smoke tests for title drag, full-row restore, and overlapping floating cards not auto-storing.

## [LRN-20260501-001] best_practice

**Logged**: 2026-05-01T00:09:30+08:00
**Priority**: high
**Status**: pending
**Area**: floating-layout

### Summary
Floating or tray-based UI recovery must be validated from persisted real state, not only from the current live widget tree.

### Details
The workbench could be hidden in `floating_tray.members` while the Dock intentionally filtered tray members, and an offscreen tray made the main workbench appear lost after restart. Recovery needs both state repair and code-level fallbacks: clamp tray coordinates, keep protected primary cards out of automatic tray grouping, and keep Dock restoration paths available for tray members.

### Suggested Action
When changing floating cards, run a smoke with a saved `floating_tray` state containing `widget-main` and offscreen coordinates, then assert the main card is visible and no longer a tray member after startup.

### Metadata
- Source: user_feedback
- Related Files: native_app/window.py, native_app/widgets/floating_tray.py
- Tags: haintag, qt, persistence, recovery

## [LRN-20260430-001] correction

**Logged**: 2026-04-30T23:47:34.5169638+08:00
**Priority**: high
**Status**: pending
**Area**: tests

### Summary
HainTag UI interaction fixes must be verified through the real visible widget/event path, not only by inspecting code or testing hidden backing editors.

### Details
The user corrected that TAG hover and right-click weight scrubbing were still unusable even though prior code-level checks passed. The visible workbench uses `_TagStreamView` and `_WorkbenchTagChip`, while the backing `TagTextEdit` is hidden, so tests against the hidden editor do not prove the user-facing interaction works.

### Suggested Action
For workbench UI fixes, add or run QtTest sequences against the visible chip/widgets: mouse hover, right-button press, move, release inside and outside the original widget, context-menu suppression, and visual screenshot capture when the change is visual.

### Metadata
- Source: user_feedback
- Related Files: native_app/widgets/output_widget.py
- Tags: haintag, qt, qtest, real_interaction
